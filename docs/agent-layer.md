## 一句话愿景

把 星核 从一个**被动军火库**(59 件装备摆那儿,用户自己翻自己挑)升级成一个**主动的「游戏 AI」**——像 RPG 里帮你练级的系统:替你**配装**、提醒你**该学新技能**、没有的技能当场帮你**锻造**。

> 题目即此:《像升级打怪一样使用 Skill》。skill=装备/技能卡,agent=帮你练级的游戏系统;不是让你自己刷,是它替你想、替你造。

## 为什么不是"加个搜索/聊天框"

- **搜索**要用户先知道关键词;Agent 层是在用户**不知道要什么**时,读懂上下文替他决定。
- **不是聊天 agent**:用户**不打字、不对话**,动作只有「点按钮」和「收通知」。Agent 焊在产品的按钮和通知背后,不是一个要去聊的助手 → 这也是它和直接用 Claude Code/Codex 的根本区别。
- **核心约束**:星核 只是 CC/Codex 的**辅助工具**,用户一整天泡在 CC/Codex 里、不看这个桌面端。所以 Agent 层必须能**在用户活在别处时触达**(见下「触达机制」)。

## 嵌入式 Agent(Pi)怎么工程化

复用开源 Pi,但对它做四件事,使它成为"住在产品里的、被驯化的" agent:

1. **限定领域** — 只懂技能管理。
2. **喂工具** — 只给:读项目文件 / 读 `~/skills/registry.json` / 读 CC·Codex 会话记录 / 装卸 skill(改 registry + 软链) / 写 SKILL.md。
3. **装 Loop** — 想→选/造→装→验,自己转圈直到完成。
4. **设边界** — 只动 `~/skills` 与已授权的项目目录;关键写操作需用户点头;不自动外发。

## 三种介入形态(= Demo 三段 = 自主性递进)

| 形态 | 游戏说法 | 接入的现有模块 | 新增能力 |
|---|---|---|---|
| **① 情境推荐**(答你所问) | 一键配装 | 项目装配 / Loadout | 项目装配界面加 **✨ AI 推荐 loadout** 按钮:Pi 读项目(代码/README/技术栈)→ 输出**有顺序的工作流**(不是散件)+ 点出用户没想到的 skill,一键装成 loadout |
| **② 前摄补刀**(想你所想) | "可学习新技能"提示 | 用量统计 / 更新管理 | 菜单栏常驻,后台读 CC/Codex 会话,发现用户**重复手动做某事**且广场有现成 skill → **弹系统通知**建议安装(可反向:建议清理冷门) |
| **③ 能力自生成**(造你所需) | 现场锻造新技能 | 安装新 Skill | 后台发现重复操作**且广场无对应 skill** → 经 hook **把建议注入 CC 会话** → 用户点「是」→ Pi 把这套操作**萃取成 skill**(写 SKILL.md + 入 registry),立即可装可用 |

## 触达机制(解决"用户不看 星核"的死穴)

两种都要做(Demo 两种都演),触达强度递进:

- **菜单栏 + 系统通知**(形态②):星核 缩进菜单栏后台常驻,读 CC/Codex 会话,对的时机弹**原生 OS 通知**——用户在 CC 里写代码也能被戳到。点通知→执行。
- **经 CC 会话冒出建议**(形态③):建议出现在用户正在用的 CC 终端里,无需切窗口。
  - ✅ **已验证机制(2026-06-30,查 CC 官方 hooks 文档确认)**:CC hook **纯被动、事件驱动**,外部进程**无法**主动 push 进活动会话。可行的近似——**星核 写 pending 文件 + CC hook 事件触发读取**:
    1. 星核 后台检测到重复操作 → 写 `~/.skillhub/pending-suggestion.json`。
    2. 用户在 CC 里下一次 **Stop / UserPromptSubmit** 触发用户配置的 hook 脚本 → 脚本读 pending 文件 → 返回 `systemMessage`(**用户终端可见**)。
    3. 用户在下一条 prompt 回 "yes" → hook(UserPromptSubmit)检测到 → 回传 星核 执行萃取。
  - **关键字段**:`systemMessage`=显示给用户(终端可见);`additionalContext`=注入给模型(用户看不到)。二者可同时返回。
  - **限制(要诚实)**:不是同步对话框,**无法阻塞等待即时回应**,是"异步提示→用户下一步回应"。对演示无碍(你控制何时触发 Stop);对产品需说明这是异步流。
  - **Codex 侧未确认**:Codex CLI 是否有等价 hook/注入机制待查;不阻塞——CC 是主演示路径。

## 与现有 PRD 的冲突点(需决断)

- PRD 现列 **Non-Goal:"不做 SKILL.md 在线编辑器(Skill 内容应在 IDE 中写)"**。形态③的"Pi 萃取生成 SKILL.md"**不违反**这条——它不是给人用的在线编辑器,是 **agent 自动生成草稿**(用户 review 后落地),属于"安装新 Skill"的智能化,不是手写编辑器。建议在 PRD 注一笔澄清,避免误读。

## Demo 必达 + 兜底(现场要求)

- **真跳 Pi**:三段各保留一个真环节(①点按钮真读项目 / ②真弹通知 / ③真萃取入库)。
- **检查点成品态**(`demo-checkpoint` 分支,铁律):① 已推荐好的工作流 loadout、② 已弹出的系统通知态、③ 已注入 CC 的建议 + 已造好的示例 skill。Pi 现场全程翻车也能靠成品态把结果演完。
- **示例数据(已确认 Yihu 真实工作流)**:整场挂在「用 Claude Code 搭的网站缺图片素材」这个普遍场景上(几乎所有人玩 coding 都会撞上)。① 段一:空网站项目 → ✨推荐 `yiqiai-image-asset-pipeline` + `kimi-webbridge` 素材工作流;② 段二:手动给网站找图调图 → 提示装 `yiqiai-image-asset-pipeline`;③ 段三:反复「`kimi-webbridge` 抓图 + `yiqiai-image-asset-pipeline` 抠透明」→ 萃取成一个**新的合成 skill**(从站点抓图并自动抠透明素材),确认该合成 skill 不在 registry;用它给空网站一把填满素材,**当场从半成品变成品**。
- 录三段干净跑通做兜底视频;彩排两遍掐表。
